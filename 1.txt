3.2.1

3.2.2内容质量评估算法


在长轨运维任务中，模型生成的执行轨迹由多步推理与多次工具调用组成，单条轨迹通常包含9–14个步骤，涉及数十次工具交互与50+参数实例，因此其质量不仅体现在最终结果是否正确，还体现在：工具调用是否合法、参数是否来源可信、跨步骤状态是否一致、推理过程是否具备训练价值

传统单指标评估（如最终正确率或单步准确率）难以全面刻画上述特征，容易出现两类典型问题：
结构性错误未被识别：轨迹表面合理但存在隐蔽错误；
高正确样本价值差异巨大：正确 ≠ 高训练价值。

为此，本项目提出一种两阶段内容质量评估算法，实现从客观正确性到训练价值的分层建模。
设一条样本轨迹为：
\mathcal{T} = {(cot_t, action_t, obs_t)}_{t=1}^{T}
其中：(cot_t)表示第t步推理文本，(action_t)表示第t步工具调用，(obs_t)表示第t步世界模型返回的工具结果，T表示轨迹长度。
内容质量评估由两阶段组成：
精确率评估体系
精确率评估用于衡量模型在给定 query 下生成的数据，是否能够正确选择工具、正确填写参数，并在多步执行过程中保持前后逻辑一致，即判断轨迹是否客观正确。该体系由4个维度、7个指标构成。

①工具正确性
该维度用于刻画模型在工具层面的基础调用可靠性，是轨迹正确性的首要约束。若工具选择或调用结构出现偏差，将直接导致后续推理链条失效。本维度包含 3 个子指标。
首先，引入工具名正确率，用于刻画模型是否选择了合法工具。遍历轨迹中所有 step的coa.action.name`字段，与预定义的 tool schema 集合进行匹配；若工具名称存在于集合中，则记为一次合法调用，否则记为非法调用。设轨迹中工具总调用次数为 (N_{call})，其中合法调用次数为 (N_{valid_tool})，则工具名正确率定义为：
Acc_{tool}=\frac{N_{valid_tool}}{N_{call}}

在工具名称合法的基础上，还需进一步验证工具调用的输入与输出结构合规性。为此，引入输入与输出 Schema 合法率，用于检测工具调用参数args以及世界模型返回的observation是否完全符合对应工具在 JSON Schema 中的定义约束（包括字段完整性、数据类型、必填项以及枚举范围等）。
设通过输入参数校验的调用次数为 (N_{action_valid})，通过输出结果校验的调用次数为 (N_{obs_valid})，则分别定义：
Acc_{action}=\frac{N_{action_valid}}{N_{call}}
Acc_{obs}=\frac{N_{obs_valid}}{N_{call}}
通过上述三项指标的联合约束，可以有效识别工具误选、参数结构错误以及返回格式异常等常见问题，从而为后续跨步骤一致性分析与过程质量评估提供可靠的结构基础。

②参数值正确性
该维度用于刻画工具调用中参数取值的来源可信性，旨在识别模型在多步执行过程中是否存在参数幻觉或不合理构造。与仅进行 Schema 合法性校验不同，本维度进一步关注参数值是否能够从任务上下文中被合理追溯。
考虑到长轨运维任务中参数来源具有明显的层次结构，本研究将参数来源划分为两类合法来源：（1）用户 query 中显式给出的实体；（2）历史步骤 observation 或当前推理文本（CoT）中可推断的信息。基于此，参数值正确性由一个综合指标进行度量，但在实现上采用两阶段来源验证机制。
首先从 query 中抽取实体集合 (\mathcal{E}_q)，包括 IP 地址、设备名、接口名等关键运维实体，随后遍历每一次工具调用中的参数字典args，对每个参数值进行纯值匹配。
对于未被判定为 query 来源的参数，本研究进一步判断其是否可以从历史执行上下文中合理推断。具体而言，对于轨迹中第t步的参数值v，构建其历史观测值集合
\mathcal{H}*t=\bigcup*{k<t}\text{Values}(obs_k)
同时获取当前步骤的推理文本 (cot_t)。若满足参数值出现在历史 observation 中，或参数值可由当前CoT合理推断（LLM评估），或参数属于预定义常量模式（如up/down、true/false 等），则认为该参数来源可信，记为正确；否则判定为参数幻觉。
Acc_{param}=\frac{N_{valid}^{param}}{N_{param}}
通过对参数来源进行细粒度建模，本维度能够显著提升对隐蔽结构错误的检测能力，为轨迹级正确性评估提供关键约束。
③跨步骤逻辑一致性
该维度用于刻画长轨执行过程中状态反馈的时序一致性。在同一条执行轨迹中，若对同一实体的同一属性进行多次 observation，且中间不存在能够导致状态变化的显式操作，则这些 observation 的取值应保持一致。该约束旨在识别模型在多步推理过程中可能出现的隐式状态漂移或观测矛盾问题。
包含同工具一致性，设总比较对数为 (N_{pair}^{same})，其中冲突对数为 (N_{conflict}^{same})，则定义
Cons_{same}=1-\frac{N_{conflict}^{same}}{N_{pair}^{same}}
以及跨工具一致性，比较来自不同工具的同名属性返回值，以检测跨工具的信息一致性
Cons_{cross}=1-\frac{N_{conflict}^{cross}}{N_{pair}^{cross}}
为轨迹级客观正确性提供关键的时序一致性保障。


④轨迹整体有效性
该维度用于刻画轨迹层面的调用冗余程度，通过统计连续或非连续地重复调用同一工具且参数相同（或经标准化后等价）的比例，评估模型是否存在无效循环或冗余查询行为。设轨迹总调用次数为 (C)，重复调用次数为 (C_{dup})
Rate_{dup}=\frac{C_{dup}}{C},
Acc_{no_dup}=1-Rate_{dup}.
在获得各子指标后，本项目对所有有效指标取算术平均，得到样本级综合精确率
Acc_{overall}=\frac{1}{K}\sum_{k=1}^{K}Acc_k,
最终，(Acc_{overall}) 作为精确率评估阶段的统一判据，用于后续阈值过滤与高价值样本筛选。


过程质量评估体系
过程质量评估体系用于衡量模型在轨迹整体正确的前提下，生成过程在逻辑连贯性、工具使用有效性、信息增益及稳定性等方面的综合质量，即判断样本是否具有较高训练价值。该体系由6个维度、7个指标构成。

①规模性
该维度用于从数据覆盖角度衡量数据集的有效规模，核心关注不同任务轨迹在求解路径上的独立性与覆盖能力。采用唯一路径数作为指标，即统计数据集中不同工具调用序列的去重数量。设数据集中共包含 (M) 条轨迹，其对应工具链签名集合为 (\mathcal{C})，则规模性指标定义为：
Scale = |\mathcal{C}|

②多样性
多样性用于刻画数据分布在宏观层面的离散程度，本研究采用归一化香农熵量化评估数据集在“个体工具调用”与“逻辑链路组合”上的离散程度，设某一分布中第 (i) 类模式出现概率为 (p_i)，则原始熵为：
H = -\sum_i p_i \log_2 p_i
进一步计算最大可能熵 (H_{max}=\log_2 N)（(N) 为模式种类数），得到归一化熵：
H_{norm}=\frac{H}{H_{max}}
熵值越接近 1，说明样本分布越均匀、逻辑冗余度越低，能够覆盖更丰富的解空间；反之则表明数据存在明显的模式集中或模板化风险。
③轨迹有效性
该维度用于从整体行为层面评估轨迹的执行质量，引入大模型作为判别器，对推理过程进行语义级综合评估。具体围绕三个核心子维度进行打分（范围 1–10）：逻辑连贯性：考察推理链路是否严密且无幻觉；工具调用有效性： 评估工具选择及参数传递的准确性；目标达成效率：衡量是否存在冗余步骤或陷入死循环。
从行为层面对轨迹可学习性进行补充建模，并计算其平均值作为轨迹有效性得分
Score_{traj}=\frac{S_{logic}+S_{tool}+S_{eff}}{3}


④专家一致性
该维度用于刻画样本评分的不确定性与置信稳定性，采用 SelectIT 指标进行量化。其核心思想是通过二级自反思机制评估模型打分的一致程度：首先在 Token 层面计算 Token-R，衡量模型对评分 token（1–5）的概率分布分离度；设基础评分为 (s_{base})，对应概率为 (p_{base})，则有：

S^{token} = S^{base} \times \frac{1}{K-1} \sum_{i=1}^{K} |P'_i - P'_{S^{base}}|
随后在句子层面引入多模板评分稳定性（Sentence-R），对同一样本使用 (K) 个不同评价 Prompt 得到 Token-R 序列，并计算：

S^{sent} = \frac{\text{Avg}\{S^{token}_i\}_{i=1}^K}{1 + \alpha \times \text{Std}\{S^{token}_i\}_{i=1}^K}
其中 (\mu) 与 (\sigma) 分别为多模板得分的均值与标准差，(\alpha) 为不确定性惩罚系数。该指标能够同时刻画评分置信度与稳定性：得分越高，说明模型判断越集中、轨迹质量越可靠。


⑤结构复杂度
该维度用于从结构规模角度刻画样本的信息承载能力，主要统计轨迹在推理深度与参数空间上的复杂程度。具体以响应步数 (T) 与参数总规模 (N_{param}) 为基础度量。

⑥ Loss Based

为从模型学习视角刻画样本难度与信息增益，该维度引入两种基于语言模型损失的度量。
首先，IFD（Instruction Following Difficulty）指标衡量模型在有指令引导下的回答损失与无指令直接预测答案损失的比值，消除答案本身长短或固有难度的影响，精准捕捉指令对答案生成的增益程度。
IFD_\theta(Q, A) = \frac{s_\theta(A|Q)}{s_\theta(A)}
其中 Observation 部分在计算中被 mask，仅对 CoT 与 Action 计入损失 。IFD 越接近 1 表示样本处于中等难度区间；显著小于 1 通常意味着样本模式已被模型充分掌握；而大于 1 则反映模型难以有效利用指令信息，属于高困惑样本。
IFD值小于0.8，说明数据简单/已掌握，指令让答案变得非常好预测。模型已经完全学会了这类逻辑。IFD值在0.8～1，说明数据中高等难度，指令提供了一定帮助。IFD值大于1，说明为高难度数据（或者逻辑性差的数据），有了指令反而让 Loss 变高了，即模型无法将指令和答案对齐，这是模型最困惑的地方。


进一步地，引入MIWV（Model-Identified Weakness Value）刻画样本对模型弱点的暴露能力。对为于样本A，通过embedding模型计算与其相似性最高的样本B，在Mask掉 Query 和 Observation部分 Loss 的前提下，分别计算模型在引入样本 B 作为前缀上下文（One-shot）和不引入上下文（Zero-shot）两种条件下，对样本A的CoT与Action的预测 Cross-Entropy Loss。
L_{\theta}(y_i | x_i) = -\frac{1}{A} \sum_{a=1}^{A} \log p(y_i^a | x_i, y_i^1...y_i^{a-1})
MIWV(x, y) = L_{\theta}(y \mid x, x_{shot}, y_{shot}) - L_{\theta}(y \mid x)
MIWV 值越大越好：给了例子 B，模型反而预测得更差了。说明模型无法利用相似案例解决当前问题，这是模型的弱点，需要保留；给了例子 B，模型Loss下降了，表现变好了，说明模型能触类旁通，这条数据相对简单或已知。MIWV 无固定上下限，核心价值在于样本间的相对排名。

为统一不同维度下的评价结果，本项目对过程质量指标进行归一化处理，并采用加权线性融合方式构建综合质量得分
Score_{final} = \sum_{i=1}^{n} w_i \cdot \hat{s}_i,
其中 (\hat{s}_i) 表示第 (i) 个指标的归一化分值，(w_i) 为对应权重系数，满足 (\sum_i w_i = 1)
综上，本项目构建了由精确率评估体系与过程质量评估体系组成的两阶段内容质量建模框架。不仅实现了对长轨数据的结构正确性判定，更建立了面向训练收益的价值建模基础，为后续质量驱动训练框架提供可量化、可解释的输入依据。



3.2.3. 基于质量评估体系的训练框架
在完成内容质量评估后，本项目进一步构建了基于数据质量导向的训练优化框架，从而实现数据质量优势向模型能力提升的系统传导。该框架围绕两条数据路径展开：一条面向通用能力提升，另一条面向长轨运维数据生成能力强化，通过统一的质量评估机制实现训练数据的结构化筛选与能力闭环优化。
首先，在通用能力数据路径上，将外部基准数据集输入过程质量评估体系，对样本在行为质量、多样性及损失敏感性等维度进行量化排序。基于综合质量得分构建高价值训练子集，并用于模型参数更新。该路径的核心目标在于提升模型在复杂工具调用、多步推理场景下的通用执行能力，为后续领域数据优化提供能力基础。
其次，在长轨运维数据路径上，采用“生成—评估—筛选—再生成”的迭代机制实现领域能力的持续优化。具体而言，首先由当前模型生成长轨运维数据，并通过精确率评估体系进行客观正确性过滤；随后在通过阈值约束的样本中，依据过程质量综合得分进行排序筛选，构建高价值领域训练子集。模型完成训练更新后，再次进行数据生成与精确率评估，从而形成能力验证与数据质量再提升的闭环过程。
本框架实现了从数据评估到模型优化的结构化闭环，使质量评估结果不能够持续转化为模型能力演进的驱动信号，为长轨智能体场景下的数据驱动训练提供系统化方法支撑。

总结

本项目构建了覆盖数据生成、内容质量评估与质量驱动训练的系统化技术框架。针对长轨数据结构复杂、跨步骤依赖强以及样本价值差异显著等特点，提出两阶段内容质量评估机制，从客观正确性与训练价值两个层面实现多维度量与统一建模，并通过综合评分机制形成可量化、可排序的数据质量刻画体系。
在此基础上，将质量评估结果嵌入数据筛选与迭代优化流程之中，实现“生成—评估—筛选—训练—再生成”的闭环机制。该方法有效打通了数据质量建模与模型能力提升之间的路径，为复杂流程任务中的大规模数据构建与模型训练提供了一定的技术支撑。
